The open-source stack behind this site

I advocate small, open-source data stacks. I had never run one end-to-end myself. So I made my own website the guinea pig.

12 July 2026 · 6 min read

For a while now I've had an opinion: most data platforms are too heavy for the problem they solve. DuckDB, Parquet, dlt and dbt can carry a real workload before you need a warehouse, a cluster, or a server someone has to babysit.

Opinions are cheap. By day I build on BigQuery — at that scale, a managed warehouse is the right call, and I'll defend that choice too. But it meant my favourite stack was something I recommended more than something I had run. That felt like the wrong order.

So this site is the experiment: the smallest complete data platform I could build, running in public, with the run logs one click away.

The whole platform

Every morning, a GitHub Action wakes up and does this:

  1. Extract & load — dlt. Pulls GitHub star counts and PyPI download history for the four tools the homepage tracks (duckdb, polars, dbt, dlt). Public APIs, no keys, no credentials to rotate. dlt lands it all in a DuckDB file under a raw schema.
  2. Transform & test — dbt. Staging views clean and dedupe; marts aggregate downloads to complete weeks and keep the daily star snapshots. Fourteen models and tests run in about five seconds, because the "warehouse" is a file.
  3. Export — DuckDB. A script runs the marts queries — including, verbatim, the SQL you see typed out on the homepage — and writes one Parquet file plus a small JSON. Then it commits both back to the repo.

Cloudflare Pages sees the commit and redeploys. And when you press Run on the homepage, that same Parquet loads into DuckDB compiled to WebAssembly and the chart queries execute in your browser — the visitor is the query engine.

That's the whole platform. No server, no scheduler, no database running anywhere at 2am. The pipeline exists for roughly a minute a day.

What actually bit me

GitHub doesn't give you history. The API tells you a repo's star count now, not last month. If you want a time series, you start saving daily snapshots and accept that your chart begins the day you did. Mine begins 11 July 2026. Downloads had no such problem — pypistats serves ~180 days back — which is why the site launched with a rich downloads chart and a star history that is still growing up.

Complete weeks only. The first version of the downloads chart had a cheerful dip at both ends — partial weeks, not real decline. The fix is one HAVING count(*) = 7 in a dbt model. Small stacks don't exempt you from data quality; they just make the fix a one-liner.

DuckDB-WASM is 34 MB. I wanted to self-host it — no third-party CDN, fully on-thesis. But the WebAssembly binary is ~34 MB uncompressed, and Cloudflare Pages caps files at 25 MiB. So it loads from jsdelivr, version-pinned, brotli'd down to ~7 MB over the wire, and only when you actually press Run. Right-sizing applies to trade-offs too: I traded a little purity for a page that stays light for everyone who never clicks.

The honesty rule was the real architecture. Early versions of this page mocked the widget with invented numbers, and the site claimed things the code didn't do yet. The rule I ended up with: every claim on the page must be either verifiable from my CV or demonstrated by the page itself. That rule deleted more copy than any design review — and it decided the build order, because the pipeline had to exist before the page could say it did. The page even polices itself now: if the pipeline ever stops committing, the "refreshed this morning" line quietly demotes itself to a dated one.

What it costs

  • GitHub Actions: €0 (public repo, ~1 minute/day)
  • Cloudflare Pages: €0
  • DuckDB, dbt, dlt: €0
  • Total: the domain name.

That's the point. Not that everything should run this way — my day job runs on BigQuery for good reasons — but that this workload fits in a web page, so anything heavier would have been costume.

Try it from your terminal

The data isn't locked inside the page — it's a public Parquet file. If you have DuckDB installed, this works from anywhere:

duckdb -c "SELECT tool, value AS stars
FROM read_parquet('https://rensevanderzee.nl/data/metrics.parquet')
WHERE mart = 'stars_current' ORDER BY stars DESC;"

A website you can query with SQL. It's a small thing, but it makes the thesis literal: the site doesn't just describe a data platform — it serves one.

What's next

A question box that translates plain English to SQL and runs it — client-side, against the same read-only Parquet, which makes it safe by construction. And a longer note on the framework behind all of this: how to pick the weight class of a data platform, from a stack like this one up to the managed warehouses I build on during the day.
